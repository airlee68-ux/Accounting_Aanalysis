## 1. 이 시스템이 **하는 일** / **하지 않는 일**

### 1-1. 제공하는 기능
- 계정과목(Chart of Accounts) 마스터 관리 — 자산/부채/자본/수익/비용
- 카테고리(수입·지출 분류) 관리 — 현업의 "관리회계용 태그"에 가까움
- 거래(Transaction) 입력·수정·삭제 — 단식부기(현금주의) 형태
- 거래 엑셀 **일괄 임포트** (템플릿 다운로드 → 미리보기 → 확정 저장)
- 재무 리포트 자동 생성:
  - 손익계산서(Income Statement) — 기간별, 카테고리별
  - 재무상태표(Balance Sheet) — 기준일 기준
  - 현금흐름표(Cash Flow) — 월별 집계
- 대시보드 — 기간별 수입·지출 총액, 월별 추이, 카테고리별 비중, 최근 거래

### 1-2. **하지 않는 일** (실무와의 간격)
회계사가 실무에서 당연히 기대할 기능이 **이 앱에는 없습니다**. 오해를 막기 위해 반드시 인지할 것:

| 실무에서 쓰는 것 | 이 앱 상태 | 참고 |
| --- | --- | --- |
| 분개(차변/대변) 입력 UI | **없음** | 데이터 모델에는 `JournalEntry`가 존재하지만, 프론트에서 입력할 수 없음 |
| 시산표(Trial Balance) | **없음** | 복식부기 데이터가 없어 의미가 약함 |
| 기말 마감(결산 분개) | **없음** | 회계기간의 경계 개념이 약함 |
| 이연·감가상각·충당부채 등 발생주의 처리 | **없음** | 현금 발생 시점에만 기록 (현금주의에 가까움) |
| 부가세 계산·신고서 | **없음** | 공급가액·세액 분리 필드 없음 |
| 법인세·원천세 | **없음** | |
| K-IFRS 재무제표 양식 (연결, 세그먼트, 주석 등) | **없음** | 개별 기본 양식의 개념적 시각화만 제공 |
| 감사추적(누가, 언제, 무엇을 수정했는가) | **없음** | |
| 사용자·권한 관리 | **없음** | 로그인 자체가 없음 — 공개 URL 접근자 누구나 수정 가능 |
| 다국어·다통화 | **없음** | KRW·한국어 고정 |
| 전자세금계산서·국세청 연동 | **없음** | |

> **결론:** 본 앱은 실제 회계법인 엔진(더존·SAP·Oracle)을 대체하지 않습니다. **회계 개념 → 데이터 흐름 → 리포트 산출**이라는 파이프라인을 손에 쥐고 돌려보며 체감하는 용도입니다.

## 2. 데이터 모델 — 실무와의 대응

### 2-1. 엔티티 구조

```
Account (계정과목)           ─┐
  · code, name               │
  · type: ASSET / LIABILITY │
          / EQUITY / REVENUE │
          / EXPENSE          │
                              │  (복식부기 확장용)
JournalEntry (분개 라인) ─────┘
  · account_id
  · debit (차변)
  · credit (대변)
  · transaction_id ───┐
                       │
Transaction (거래) ◄───┘
  · date, description, amount
  · type: INCOME / EXPENSE / TRANSFER
  · category_id
  · memo

Category (카테고리)
  · name, type (INCOME/EXPENSE)
```

### 2-2. 실무 용어 ↔ 본 앱 매핑

| 실무 개념 | 이 앱의 대응 | 비고 |
| --- | --- | --- |
| 계정과목표(CoA) | `Account` | 코드·명·유형(자산/부채/자본/수익/비용) |
| 보조부(거래 원장) | `Transaction` | 단식 거래 단위 |
| 분개장/원장 | `JournalEntry` | **모델만 존재**, UI 없음 |
| 관리회계 분류/코스트센터 | `Category` | 자유로운 수입·지출 분류 태그 |
| 전표 헤더 ↔ 라인 | `Transaction` ↔ `JournalEntry` | 1:N 관계로 확장 가능 |

### 2-3. 기본 시드 데이터
설치 시 [`backend/app/seed.py`](../backend/app/seed.py)가 기본 계정과목 16개(1010 현금 ~ 5060 소모품비)와 카테고리 10개(제품매출, 급여 등)를 넣어두고, 샘플 거래를 자동 생성합니다. **이는 데모용**이며 실제 회계에서 쓰는 계정과목표(예: 한국채택 국제회계기준의 표준 CoA)와 **다를 수 있음**을 인지할 것.

## 3. 리포트 산출 방식 — 회계사 관점에서 꼭 알아야 할 것

### 3-1. 손익계산서 (Income Statement)
- 구현: [`routers/reports.py::income_statement`](../backend/app/routers/reports.py)
- 로직: 지정 기간 내 `Transaction`을 `Category × type(INCOME/EXPENSE)` 기준으로 합산.
- **회계사가 주의할 점:**
  - **단층 구조**입니다. 매출 → 매출원가 → 매출총이익 → 판관비 → 영업이익 → 영업외손익 → 법인세차감전 → 당기순이익 같은 **다층 IS 구조를 재현하지 않습니다.**
  - 단순히 `total_revenue - total_expense = net_income`.
  - 매출원가/판관비 구분도 "카테고리 이름"으로만 가능 (코드상 자동 분류 없음).

### 3-2. 재무상태표 (Balance Sheet)
- 구현: [`routers/reports.py::balance_sheet`](../backend/app/routers/reports.py)
- **두 가지 모드로 동작**합니다:
  1. **복식부기 모드** — `JournalEntry` 데이터가 존재하면, 계정과목별 차변·대변을 집계해 정상적인 BS 생성. (자산·비용: 차변-대변, 부채·자본·수익: 대변-차변)
  2. **단식부기 간이 모드** — 분개가 없으면 기본값. 전 기간 수입 합계를 **현금성 자산**으로, 같은 금액을 **이익잉여금**으로 잡는 **2줄짜리 단순 BS**를 만듭니다.
- **회계사가 주의할 점:**
  - 현재 프로덕션은 **단식 간이 모드로만 동작 중**입니다 (분개 UI가 없어 `JournalEntry`가 비어있음).
  - 즉, 지금 보이는 BS는 "자산 = 자본, 부채 = 0"인 **축약형 BS**이며, 실제 BS 구조와 다릅니다.
  - 매입채무·미지급금·차입금·재고자산 등은 **거래로는 잡히지만 BS에는 반영되지 않습니다.**

### 3-3. 현금흐름표 (Cash Flow)
- 구현: [`routers/reports.py::cash_flow`](../backend/app/routers/reports.py)
- 로직: 월별로 `INCOME` 합계(inflow) / `EXPENSE` 합계(outflow) / 순증감(net).
- **회계사가 주의할 점:**
  - **영업활동·투자활동·재무활동 구분이 없습니다.** (K-IFRS 현금흐름표 형식과 다름)
  - 간접법·직접법 구분도 없습니다. 단순히 **"월별 현금 in/out"**입니다.
  - 회계사에게 익숙한 "당기순이익 → 비현금항목 가감 → 영업활동현금흐름"의 간접법 전개는 **구현되어 있지 않습니다.**

### 3-4. 대시보드
- 기간 내 총수입·총지출·순액
- 월별 추이 (막대)
- 카테고리별 비중 (도넛)
- 최근 거래 10건

## 4. 권장 워크플로우 (예비 회계사용)

### Step 1. 계정과목 확인·보강
- **계정과목** 메뉴에서 기본 CoA 확인.
- 입사 예정 회계법인의 감사 의뢰인 업종 CoA로 바꾸고 싶다면, 계정과목을 직접 추가/편집.

### Step 2. 카테고리 설계
- 관리회계 관점으로 카테고리를 먼저 설계 (예: `제품매출`, `용역매출`, `인건비`, `임차료`, `판촉비` 등).
- 카테고리는 "이 거래가 어느 손익 항목으로 집계되는가"의 키가 됩니다.

### Step 3. 거래 입력 (두 가지 방법)
- **(A) 수동 입력** — 거래 페이지에서 날짜·구분·내용·금액·카테고리·메모 입력.
- **(B) 엑셀 일괄 임포트** — 권장
  1. **[템플릿 다운로드]** 클릭 → `transactions_template.xlsx` 저장
  2. 엑셀 파일에 데이터 채우기. **필수 컬럼: `날짜 / 구분 / 내용 / 금액 / 카테고리 / 메모`**
     - 날짜 형식: `YYYY-MM-DD` / `YYYY/MM/DD` / `YYYY.MM.DD` / `YYYYMMDD`
     - 구분: `수입` 또는 `지출` (또는 `INCOME`/`EXPENSE`)
     - 금액: 숫자 (쉼표·"원" 허용)
     - 카테고리: **기존 카테고리명과 정확히 일치**해야 자동 매칭됨 (다르면 오류 표시)
  3. **[엑셀 업로드]** → 미리보기에서 유효/오류 행 확인 → **[N건 저장]**

### Step 4. 리포트 열람
- 기간을 바꿔가며 **손익계산서 · 재무상태표 · 현금흐름표**에 어떻게 반영되는지 관찰.
- 이 과정을 통해 **"거래 → 집계 → 재무제표"** 흐름의 직관을 얻는 것이 본 앱의 핵심 가치입니다.

## 5. 교육적 활용 시나리오 제안

예비 회계사가 본 앱으로 연습하면 좋은 시나리오 예시:

1. **손익 구조 체감** — 같은 금액의 "매출 100만원"이라도 카테고리를 `제품매출` vs `영업외수익`으로 바꿔보며 IS의 라벨이 어떻게 달라지는지.
2. **월별 현금 흐름 분석** — 12개월치 가상 거래를 임포트해, 성수기/비수기 월별 현금흐름 패턴을 도출.
3. **간이 BS의 한계 체감** — 차입금 거래를 입력해도 BS에 부채로 잡히지 않는 현상을 직접 확인하고, 복식부기가 왜 필요한지 체득.
4. **카테고리 재설계 실습** — 동일 거래 집합을 두고 카테고리 체계를 달리 짜서, 관리회계 보고서가 달라지는 경험.

## 6. 보안·데이터 경고

- **이 앱은 공개 URL 상에서 인증 없이 동작합니다.** 입력하는 모든 데이터는 누구나 볼 수 있다고 가정해야 합니다.
- **실제 회계법인 의뢰인의 실 거래·개인정보는 절대 입력하지 마세요.** 반드시 **가상 데이터 또는 교육용 샘플 데이터**만 사용.
- 개인 실습이 필요하면 레포를 클론 후 **로컬 환경(localhost)**에서 구동하는 것을 권장.

## 7. 확장 로드맵 (언젠가 추가 가능)

- 분개 입력 UI (차변·대변 직접 기록) → 시산표 → 제대로 된 복식부기 BS
- 부가세 공급가액·세액 분리 + 부가세 신고 보조

- 현금흐름표 영업/투자/재무 활동 구분 (간접법)
- 감사추적(audit log) + 사용자·권한
- 다통화 + 환율 변환
